## Repository Layout

```
knova/
├── backend/     FastAPI service (Python 3.12, SQLAlchemy, Alembic, PostgreSQL + pgvector)
├── frontend/    Next.js app (TypeScript, Tailwind CSS)
└── ml/          Models, notebooks, and data for the recommendation engine
```

Each area has its own dependencies and tooling. Please keep changes scoped to the
area they affect where possible.

## Getting Started

### Backend

See [`backend/README.md`](backend/README.md) for full instructions. In short:

```bash
cd backend
python3.12 -m venv .venv
source .venv/bin/activate
uv sync          # or: pip install -r requirements.txt
cp .env.example .env
alembic upgrade head
uvicorn main:app --reload
```

The API docs are served at `http://localhost:8000/api/v1/docs`.

### Frontend

```bash
cd frontend
npm install
npm run dev
```

The app runs at `http://localhost:3000`.

## Branching Strategy

- `main` is the stable, deployable branch. Do not commit to it directly.
- Create a feature branch off `main` for every change:

  ```bash
  git checkout -b feat/topic-recommendations
  git checkout -b fix/auth-token-refresh
  ```

- Use short, descriptive, kebab-case branch names prefixed by type
  (`feat/`, `fix/`, `chore/`, `docs/`, `refactor/`, `test/`).

## Commit Messages

Follow the [Conventional Commits](https://www.conventionalcommits.org/) style,
consistent with the existing history:

```
<type>: <short summary in the imperative mood>
```

**Types**

| Type       | When to use                                            |
|------------|--------------------------------------------------------|
| `feat`     | A new feature                                          |
| `fix`      | A bug fix                                              |
| `update`   | An incremental change to existing functionality        |
| `refactor` | Code change that neither fixes a bug nor adds a feature|
| `docs`     | Documentation only                                     |
| `test`     | Adding or updating tests                               |
| `chore`    | Tooling, config, dependencies, or maintenance          |

**Examples**

```
feat: Add topic create and list endpoints
fix: Handle expired refresh tokens on /auth/refresh
docs: Document environment variables in backend README
```

Keep the summary under ~72 characters and add a body when more context helps.

## Coding Standards

### Backend (Python)

- Target **Python 3.12+**.
- Follow [PEP 8](https://peps.python.org/pep-0008/); use type hints throughout.
- Prefer `async`/`await` for I/O — the stack is async (FastAPI + asyncpg + SQLAlchemy).
- Keep business logic out of route handlers; use the existing `core/`, `src/db/`,
  and `src/auth/` structure.
- Validate request/response payloads with Pydantic models.
- Never commit secrets. Configuration belongs in `.env` (which is gitignored).

### Frontend (TypeScript / Next.js)

- Use **TypeScript** for all new code.
- Run the linter before pushing:

  ```bash
  npm run lint
  ```

- Follow the App Router conventions under `src/app/`, and place shared UI in
  `src/components/`.
- Style with Tailwind CSS.

## Database Migrations

Any change to SQLAlchemy models must be accompanied by an Alembic migration:

```bash
# Generate a migration from model changes
alembic revision --autogenerate -m "add topic table"

# Apply it
alembic upgrade head

# Roll back one step
alembic downgrade -1
```

Review the autogenerated migration before committing it — autogeneration is not
always complete. Migrations must be committed alongside the model changes.

## Testing

- Add or update tests for the behavior you change.
- Ensure the backend starts and existing migrations apply cleanly.
- Ensure the frontend builds (`npm run build`) and lints without errors.

Please describe how you verified your change in the pull request.

## Pull Requests

Before opening a PR:

1. Rebase or merge the latest `main` into your branch.
2. Ensure your branch builds, lints, and (where applicable) tests pass.
3. Keep PRs focused — one logical change per PR.

Your PR description should include:

- **What** changed and **why**.
- Any related issues (e.g. `Closes #123`).
- Screenshots for UI changes.
- Notes on migrations, config, or environment variable changes.

A maintainer will review your PR. Address feedback by pushing additional commits
to the same branch.

---

## Reporting Issues

When filing an issue, please include:

- A clear description of the problem or request.
- Steps to reproduce (for bugs), including expected vs. actual behavior.
- Environment details (OS, Python/Node version, browser) where relevant.
- Relevant logs or screenshots.
